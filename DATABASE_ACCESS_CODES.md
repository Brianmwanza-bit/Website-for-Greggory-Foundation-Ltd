# Database-Based Access Codes System

## Overview
Admin and Developer access codes are now stored in the database instead of environment variables. This provides:
- **Better security** - Codes are hashed and tracked
- **Usage limits** - Can set max uses per code
- **Expiration dates** - Codes can expire automatically
- **Audit trail** - Track who uses which code
- **Dynamic management** - Add/remove codes without restarting server

## Database Table: `access_codes`

### Schema
```sql
CREATE TABLE access_codes (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    code_type ENUM('admin', 'developer', 'super_admin') NOT NULL,
    code_value VARCHAR(255) NOT NULL UNIQUE,
    code_hash VARCHAR(255) NOT NULL, -- bcrypt hashed
    is_active BOOLEAN DEFAULT TRUE,
    expires_at TIMESTAMP NULL DEFAULT NULL,
    max_uses INT DEFAULT NULL, -- NULL means unlimited
    current_uses INT DEFAULT 0,
    created_by BIGINT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    deleted_at TIMESTAMP NULL DEFAULT NULL
);
```

### Default Codes (Auto-Created)
| Type | Code | Status |
|------|------|--------|
| admin | ADMIN123 | Active, unlimited uses |
| developer | DEV123 | Active, unlimited uses |

### Access Code Management (Admin Only)

#### List All Codes
```
GET /api/access-codes
Response: {
  "success": true,
  "codes": [
    {
      "id": 1,
      "code_type": "admin",
      "code_value": "ADMIN123",
      "is_active": true,
      "expires_at": null,
      "max_uses": null,
      "current_uses": 5,
      "created_at": "2024-01-15 10:30:00"
    }
  ]
}
```

#### Create New Code
```
POST /api/access-codes
Body: {
  "code_type": "admin",        // or "developer" or "super_admin"
  "code_value": "NEWCODE2024",
  "expires_at": "2024-12-31",  // optional
  "max_uses": 10              // optional, null for unlimited
}
```

#### Update Code (Activate/Deactivate)
```
PUT /api/access-codes/:id
Body: {
  "is_active": false,          // deactivate
  "expires_at": "2024-06-30",  // change expiry
  "max_uses": 5               // change max uses
}
```

#### Delete Code (Soft Delete)
```
DELETE /api/access-codes/:id
```

## SQL Commands

### Add New Admin Code
```sql
-- Generate hash first (use bcrypt with 10 rounds)
-- For code "MYADMIN2024", hash would be generated by Node.js

INSERT INTO access_codes (code_type, code_value, code_hash, is_active, max_uses)
VALUES (
  'admin',
  'MYADMIN2024',
  '$2a$10$...',  -- bcrypt hash of 'MYADMIN2024'
  TRUE,
  NULL  -- unlimited uses
);
```

### Add New Developer Code
```sql
INSERT INTO access_codes (code_type, code_value, code_hash, is_active, max_uses)
VALUES (
  'developer',
  'MYDEV2024',
  '$2a$10$...',  -- bcrypt hash
  TRUE,
  50  -- max 50 uses
);
```

### View Active Codes
```sql
SELECT * FROM active_access_codes;
```

### Deactivate a Code
```sql
UPDATE access_codes 
SET is_active = FALSE 
WHERE code_value = 'ADMIN123';
```

### Check Code Usage
```sql
SELECT 
  code_type,
  code_value,
  current_uses,
  max_uses,
  CASE 
    WHEN max_uses IS NULL THEN 'unlimited'
    ELSE CONCAT(current_uses, '/', max_uses)
  END as usage
FROM access_codes 
WHERE deleted_at IS NULL;
```

## Authentication Flow

### User Types and Their Tables

| User Type | Login Endpoint | User Table | Code Table | Code Type |
|-----------|---------------|------------|------------|-----------|
| **Regular User** | `/api/login` | `users` | N/A | N/A |
| **Admin** | `/api/admin/authenticate` | `admin_users` | `access_codes` | `admin` |
| **Developer** | `/api/developer/authenticate` | `developer_users` | `access_codes` | `developer` |

### Login Process

1. **User submits credentials** (email + password + access code)
2. **Server validates access code** against `access_codes` table
   - Checks if code exists
   - Checks if code is active
   - Checks if code has expired
   - Checks if max uses reached
   - Increments usage counter
3. **Server validates user** from appropriate user table
4. **Server returns JWT token** on success

## Code Features

### Usage Tracking
Each time a code is used successfully:
- `current_uses` increments by 1
- If `max_uses` is set and reached, code becomes invalid

### Expiration
- Set `expires_at` to a future date
- Code automatically becomes invalid after that date
- NULL means never expires

### Soft Delete
- Codes are soft deleted (not permanently removed)
- Set `deleted_at` to mark as deleted
- Can be restored by setting `deleted_at = NULL`

## Troubleshooting

### Code Not Working
```sql
-- Check code status
SELECT 
  code_value,
  is_active,
  expires_at,
  max_uses,
  current_uses,
  CASE 
    WHEN is_active = FALSE THEN 'inactive'
    WHEN expires_at IS NOT NULL AND expires_at < NOW() THEN 'expired'
    WHEN max_uses IS NOT NULL AND current_uses >= max_uses THEN 'depleted'
    ELSE 'active'
  END as status
FROM access_codes 
WHERE code_value = 'ADMIN123';
```

### Reset Code Usage
```sql
-- Reset usage counter
UPDATE access_codes 
SET current_uses = 0 
WHERE code_value = 'ADMIN123';
```

### View All Codes
```sql
SELECT 
  id,
  code_type,
  code_value,
  is_active,
  max_uses,
  current_uses,
  created_at
FROM access_codes 
WHERE deleted_at IS NULL
ORDER BY code_type, created_at DESC;
```

## Security Notes

1. **Code Values Stored Plaintext** - For lookup purposes (could be hashed if needed)
2. **Code Hashes Stored** - For additional verification (currently unused but available)
3. **Usage Tracking** - Every successful use is logged
4. **IP Restriction** - Still enforced separately from code validation
5. **Rate Limiting** - Still applies to prevent brute force

## Migration from Environment Variables

If you have existing codes in `.env`:

```sql
-- Add your existing env codes to database
INSERT INTO access_codes (code_type, code_value, code_hash, is_active)
VALUES 
  ('admin', 'YOUR_OLD_ADMIN_CODE', '$2a$10$...', TRUE),
  ('developer', 'YOUR_OLD_DEV_CODE', '$2a$10$...', TRUE);
```

Then remove from `.env`:
```diff
- ADMIN_ACCESS_CODE=YOUR_OLD_ADMIN_CODE
- DEV_CODE=YOUR_OLD_DEV_CODE
```
